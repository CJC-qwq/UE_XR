# 需求转功能流程（专业 Agent Team）

## 1. 目标

将需求稳定转化为可验证的 Unreal 项目改动，减少理解偏差、返工和集成风险。

## 2. 执行原则

* 需求不清晰时必须先追问，不对高影响改动做主观猜测。
* 超过单点小修的任务，采用专业 agent team 并行分工。
* 每次交付后必须更新维护进度文档。
* 规划阶段应用 tacit knowledge，结合上下文、历史决策、引擎约束、插件边界和真实使用场景识别隐性限制。
* 优先沿用现有 Unreal 模块、插件、配置和 Blueprint/C++ 边界，不随意扩散修改面。

## 3. 标准流程

1. 需求接收与歧义扫描
* 识别目标、范围、验收标准、边界条件、异常路径和影响模块。
* 判断改动属于主工程、插件、配置、资源还是文档流程。

2. 追问澄清
* 对不明确项做最小必要追问。
* 输出可执行定义，包括 expected behavior、exception cases、priority 和 definition of done。

3. Agent Team 分工
* 产品/流程 agent：整理需求口径、验收标准和文档更新点。
* 工程 agent：处理 C++ 模块、插件、配置、接口或运行时逻辑。
* 体验 agent：关注 Blueprint 暴露、编辑器交互、资源接入和使用路径。
* QA agent：整理验证清单、回归范围和未验证风险。

4. 科学实施
* 先建立最小可验证闭环，再扩大实现范围。
* 每一步都保留可观察结果，例如编译结果、运行日志、编辑器加载状态或手工验证记录。

5. 验证与回归
* 优先执行受影响范围内最小必要验证。
* 至少覆盖关键链路和异常链路。
* 若条件不足无法验证，必须在维护记录中明确写出未验证项与原因。

6. 文档与进度更新
* 更新维护进度记录。
* 当流程、运行方式、构建方式或模块职责变化时，更新相关设计或操作文档。

## 4. 输出模板

* 需求摘要
* 关键澄清点
* 实施清单（按模块）
* 风险与回退策略
* 验证结果
* 下一步计划
